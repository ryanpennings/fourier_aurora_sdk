# Simulation

The `sim/` directory contains a MuJoCo-based simulator that stands in for the real robot when the Aurora server is configured with `RunType: 0`. It is intended to run inside the Aurora Docker container, where the robot models and terrains are pre-installed.

## Components

```
sim/
├── start_simulate_root.py      # Interactive launcher: pick robot + terrain, spawn viewer
└── mujoco/
    ├── viewer.py               # Modified MuJoCo interactive viewer with an LCM bridge
    ├── simulator_pub_msg.py    # LCM message: simulator → controller (auto-generated)
    ├── simulator_sub_msg.py    # LCM message: controller → simulator (auto-generated)
    └── get_names_and_id.py     # Utility: print joint/link names + IDs from a URDF
```

## Launching the simulator

From inside the container:

```bash
python3 sim/start_simulate_root.py
```

The launcher:

1. Lists robot models — every subdirectory of `/usr/local/resources` (excluding `terrains`) — and prompts for a selection **by number**.
2. Lists terrains — every `.xml` file under `/usr/local/terrains` — and prompts for a selection by number.
3. Resolves the robot MJCF at `/usr/local/resources/<ROBOT>/mjcf/main.xml` and runs:

   ```bash
   /usr/bin/python3 sim/mujoco/viewer.py --mjcf <robot_main.xml> --terrain <terrain.xml>
   ```

4. Streams the viewer's stdout/stderr to your terminal. Ctrl-C terminates the viewer process.

The prompts are printed in Chinese; enter the number of your choice at each prompt.

### Terrain merging

`viewer.py` does not load the robot and terrain files separately. Its loader parses both XML files and appends each terrain element into the matching section of the robot MJCF, writes the merged model as `main_with_terrain.xml` next to `main.xml`, and loads that combined file into MuJoCo.

## The viewer (`sim/mujoco/viewer.py`)

This is DeepMind's interactive MuJoCo viewer (Apache 2.0) extended with an **LCM bridge** in the physics loop. Every physics iteration, while the simulation is running (unpaused), it:

1. **Publishes robot state** (`simulator_state_pub`) on LCM channel **`msg_simulator_to_controller`**.
2. **Applies actuator commands** (`apply_control`) from the most recent message on LCM channel **`msg_controller_to_simulator`** (subscribed on a background thread).
3. Steps physics with MuJoCo's usual real-time synchronization (speed control, control-noise injection, pause/resume all still work from the GUI).

There is also a disabled `motion_cap` hook that, if re-enabled, appends base pose + joint positions to `motion_data.txt` at 120 Hz.

### State message: `simulator_pub_msg`

| Field | Type | Contents |
|---|---|---|
| `timestamp` | int64 | Wall-clock time in ms |
| `motorNum` | int32 | Number of actuators (`model.nu`) |
| `q`, `qDot` | double[motorNum] | Joint positions and velocities (from MuJoCo joint sensors) |
| `acc` | double[3] | Base accelerometer (`baselink-baseAcc` sensor) |
| `omega` | double[3] | Base gyro (`baselink-gyro` sensor) |
| `quat` | double[4] | Base orientation (`baselink-quat` sensor, MuJoCo w-x-y-z order) |
| `lf_touch`, `rf_touch` | double | Left/right foot touch sensors |
| `pos`, `vel` | double[3] | Base position and linear velocity |

The sensor layout is read positionally from `data.sensordata`: joint positions first (`model.nu` values), then joint velocities, then the base sensors located by name (`baselink-baseAcc`, `baselink-gyro`, `baselink-quat`), so robot MJCF files must define sensors in this arrangement.

### Command message: `simulator_sub_msg`

| Field | Type | Contents |
|---|---|---|
| `timestamp` | int64 | Command timestamp |
| `motorNum` | int32 | Number of actuators |
| `started` | int8 | 0 = output zero torque; 1 = apply the control law |
| `qCmd`, `qDotCmd` | double[motorNum] | Desired joint positions and velocities |
| `torqCmd` | double[motorNum] | Feed-forward torque |
| `q_kp`, `q_kd` | double[motorNum] | Per-joint PD gains |

The applied control law is a joint-space PD with torque feed-forward:

```
ctrl = q_kp * (qCmd − q) + q_kd * (qDotCmd − qDot) + torqCmd
```

If no command has been received yet, or `started == 0`, all actuator commands are zero.

Both message classes are auto-generated by the LCM tool ("DO NOT MODIFY BY HAND"); regenerate them from the `.lcm` definitions if the schema changes.

## `get_names_and_id.py`

A standalone utility (uses **Pinocchio**, not MuJoCo) that loads a URDF and prints every joint ID/name and frame (link) ID/name. Useful for mapping control-group joint order to actual robot joints. Edit the hardcoded path at the bottom before running:

```python
urdf_file_path = './resources/GR2T2V2/urdf/GR2T2v2.urdf'
```

```bash
python3 sim/mujoco/get_names_and_id.py
```

## How it fits together

Note that the client SDK never talks to the simulator directly:

```
AuroraClient  ──DDS──►  Aurora server  ──LCM──►  MuJoCo viewer
   (your code)           (RunType: 0)             (physics + GUI)
```

The Aurora server consumes the simulator's state message exactly as it would real sensor data, and its low-level joint commands become the `simulator_sub_msg` PD targets. This means every client-side example works identically against simulation and hardware — only the server's `RunType` changes.
